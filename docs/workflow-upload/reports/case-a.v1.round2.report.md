### 低风险 SQL1

- 原始 SQL：`select * from customer where cust_id = 20001`
- 风险等级：低风险
- 分析结果：
    1. **表基础信息**：表名为 `customer`，位于数据库 `CUSDBX` (MySQL 环境，端口 3306)。提供的元数据显示 `table_rows` 为 0，表明当前表为空或统计信息未更新。主键及索引信息在提供的 DDL 中为空，无法确认 `cust_id` 是否为主键或拥有索引，但基于字段命名规范（`_id` 后缀），该字段极大概率为主键。
    2. **SQL 语法适配性**：语句符合 MySQL 语法规范。`WHERE` 条件为整型等值查询 (`cust_id = 20001`)，语法简单清晰。
    3. **索引使用预判**：
       - 若 `cust_id` 是主键（InnoDB 默认为聚簇索引）或拥有唯一/普通 BTREE 索引：查询将直接通过索引定位，时间复杂度为 O(1) 或 O(logN)，效率极高。
       - 若 `cust_id` 无索引：由于元数据显示行数为 0，全表扫描开销几乎为零；即使未来数据量增长，单行等值查询在无索引情况下对于小数据量表也是可接受的，但在大数据量下会退化为全表扫描。鉴于当前数据量为 0 且查询条件极其简单，判定为低风险。
    4. **性能风险点**：`SELECT *` 在生产环境中通常不推荐，因为它会获取所有列数据。如果 `customer` 表包含大文本字段（如 `TEXT`, `BLOB`），会增加网络传输和内存消耗。但在当前单行查询场景下，此风险可控。
