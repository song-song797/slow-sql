### 中风险 SQL1

- 原始 SQL：`select count(*) as total_count from audit_log where created_at >= '2026-03-01 00:00:00' and created_at < '2026-03-21 00:00:00' and request_path like '/api/order/%'`
- 风险等级：中风险
- 分析结果：
    1. **表基础信息**：表名为 `audit_log`，当前元数据显示行数为 0（生产环境通常审计日志表为大表，行数极易超过百万级）。未提供 DDL，默认主键情况未知，但审计表通常以自增 ID 为主键。
    2. **SQL 语法适配性**：语法符合 MySQL 规范。`COUNT(*)` 在 MySQL InnoDB 引擎下优化较好，但依赖过滤条件的效率。
    3. **索引使用预判**：
       - **关键缺失**：提供的元数据中 `ddl` 为空，且未列出任何索引信息（Indexes structure）。根据规则，必须判定为 **该字段无对应索引**。
       - **过滤条件分析**：
         - `created_at` 范围查询：若无索引，将导致全表扫描。
         - `request_path` 模糊查询：`LIKE '/api/order/%'` 为左匹配模式，若有索引可利用最左前缀原则；但若无索引，则需逐行扫描判断。
       - **执行计划推演**：由于缺乏 `created_at` 或 `request_path` 的索引，MySQL 优化器将被迫进行 **全表扫描 (ALL)**。对于审计日志这类通常数据量巨大的表，全表扫描计算 `COUNT` 会带来极高的 IO 开销和 CPU 消耗。
    4. **性能风险点**：
       - 若 `audit_log` 表数据量达到中表（10 万行）或大表（>100 万行）级别，此查询在无索引情况下耗时将从毫秒级飙升至秒级甚至超时。
       - `LIKE` 查询虽然使用了前缀匹配，但在没有索引支撑的情况下，其优化效果等同于全表扫描中的字符串比较操作。

| 索引缺失 | 高 | 1. 建议为高频查询条件字段添加复合索引。<br>- 针对 SQL 1: 由于查询包含时间范围过滤和前缀模糊匹配，建议创建复合索引以覆盖过滤条件。<br>执行语句 (MySQL): `CREATE INDEX idx_audit_log_created_path ON audit_log (created_at, request_path) USING BTREE;`<br>**说明**：将 `created_at` 放在索引首位以支持范围查询，`request_path` 放在第二位。虽然 `request_path` 后是范围/模糊查询可能导致第二位索引利用率下降，但在 MySQL 5.6+ 中仍可利用索引进行过滤（Index Condition Pushdown），显著减少回表次数。若时间范围跨度极大，单独对 `request_path` 建索引可能更优，需根据实际数据分布测试。 |
| SQL 写法不当 | 低 | 2. 避免在业务高峰期对大表进行无索引的全表 `COUNT` 操作。<br>- 针对 SQL 1: 如果仅需估算数量，可考虑使用 `EXPLAIN` 获取 rows 估算值，或引入 Redis 等缓存层统计每日/每时段日志量，避免直接查库。 |
